# 설계 철학

> Flow OS = 뇌의 약점을 보완하는 외부화된 인지 시스템 (Externalized Cognition System)

## 핵심 원칙

```
[ Cognitive Relief ] → [ Frictionless Return ] → [ Instant Resumption ]
[ 인지 부담 경감 ]   → [ 마찰 없는 복귀 ]      → [ 즉시 재개 ]
```

1. **Cognitive Relief (인지 부담 경감)**: 환경 설정에 낭비되는 정신 에너지를 보존하여, 실제 작업 수행에만 집중할 수 있도록 한다.
2. **Frictionless Return (마찰 없는 복귀)**: 산만해진 후 돌아오는 장벽을 극단적으로 낮춘다. 이탈을 완전히 막을 수는 없지만, 복귀를 쉽게 만들면 포기 대신 진전이 생긴다.
3. **Instant Resumption (즉시 재개)**: 작업 흐름의 정확한 진입점을 기록한다. 마찰이 제거되면, 어떤 순간에도 즉시 이어서 시작할 수 있다.

## 설계 원칙

1. **기억에 의존하지 않는다**: 필요한 정보를 머릿속이 아니라 시스템에 기록하고 직접 본다.
2. **시작점은 항상 동일하다**: 어떤 환경이든 터미널 진입점이 같아 시작이 쉽다.
3. **Remote가 기준이다**: 기준 상태는 항상 Remote(GitHub)에 보관한다.
4. **행동이 명확해야 한다**: 다음 첫 걸음을 망설임 없이 내디딜 수 있게 돕는다.

## 구조적 분리

Flow OS는 시스템과 실제 작업 환경을 명확히 분리하는 **4계층 구조**를 사용합니다.
 동시에, 단순한 명령 기반 시스템이 아니라 **상태 기반 시스템(State-driven system)** 으로 동작합니다.

> 시스템은 현재 상태를 기록하고,
>  사용자는 그 상태를 보고 즉시 행동합니다.

- **OS 레이어**: Flow OS 자체의 실행 환경과 설정
- **데이터 레이어**: 사용자의 실제 프로젝트와 작업물
- **상태 레이어**: 작업 진행 상태와 핵심 메타 정보
- **의도 레이어:** 디렉토리와 파일의 저장소를 의도로 분류

설치가 완료되면 홈 디렉토리 `~` 아래에 다음과 같은 구조가 자동으로 생성됩니다.

```
~/
├── (OS의 기본 디렉토리)
├── .flow-os/      # OS 레이어
└── my/            # 데이터 레이어
    ├── .flow/     # 상태 레이어
    └── (purpose)  # 의도 레이어
```

`.`으로 시작하는 디렉토리는 hidden 처리되므로, 사용자 입장에서는 사실상 하나의 작업 공간만 보이게 됩니다. 

```bash
~/
├── (OS의 기본 디렉토리)
└── my
```

예를 들어 macOS에서 다음과 같이 `my`가 하나의 작업 공간으로 자연스럽게 인식됩니다.

```bash
~/
├── Applications
├── Desktop
├── Documents
├── Downloads
├── Library
├── Movies
├── Music
├── my
├── Pictures
└── Public
```

즉, 사용자는 별도의 시스템 구조를 의식하지 않고  `my` 디렉토리를 중심으로 작업을 시작할 수 있으며, **명령을 기억하기보다, 현재 상태를 보고 다음 행동을 결정**하게 됩니다.

## 개발자 친화적 접근

고급 사용자나 터미널 환경에 익숙한 사용자를 위해 Flow OS는 내부 레이어로 빠르게 이동할 수 있는 alias를 제공합니다.

예를 들어:

- `fos` → `~/.flow-os`   (OS 레이어로 즉시 이동)
- `my` → `~/my`                  (데이터 레이어로 즉시 이동)
- `flow` → `~/my/.flow` (상태 레이어로 즉시 이동)
- **의도 레이어:**

이로써

- 일반 사용자에게는 단순한 Workspace 경험을 제공하고
- 고급 사용자에게는 시스템 내부를 손쉽게 다룰 수 있는 유연성을 제공합니다.

이 구조를 통해 Flow OS는 **단순함과 확장성, 그리고 제어 가능성**을 동시에 확보합니다.

## 구조 단순화의 진화 (Design Evolution)

현재의 구조는 처음부터 단순했던 것이 아니라, 여러 실험을 거쳐 **의도적으로 압축된 결과**입니다.

초기에는 다음과 같은 구조를 사용했습니다.

```
~/
├── (OS의 기본 디렉토리)
├── flow-entry       # 몰입 진입 단계 (Focus Layer)
├── flow-state       # 몰입 실행 상태 (State Layer)
├── my               # 작업 공간
├── .flow-os         # 시스템 코어
```

이 구조는 몰입 단계에 진입하는 과정을 구분하므로 개념적으로는 명확했지만, 사용자 입장에서는 **구조가 과도하게 노출되고 진입점이 분산되는 문제**가 있었습니다.

이후 구조는 점차 확장되며 복잡도가 증가했습니다:

```
~/my
~/flow-entry
~/flow-state
~/.flow-os
~/.flow
~/.flow-architecture
```

이 과정에서 도출된 핵심 인사이트는

> **구조를 드러내기 보다, 단순하게 응축할수록 인지적 부담을 덜 수 있어서 더 쉬워진다.**

### 최종 구조의 의미

Flow OS는 구조를 다음과 같이 재정의합니다.

- 외부에 드러나는 것은 단 하나 → `my`
- 나머지는 내부 레이어로 흡수 → `.flow`, `.flow-os`

즉,

```
다수의 디렉토리 → 3계층 구조 → 사용자에게는 1개의 Workspace
```

라는 변화는 단순한 정리를 넘어,

- **인지 부하 감소**
- **작업 흐름 유지 (Flow 유지)**
- **상태 중심 사고 강화**
- **시스템과 사용자 영역의 분리**

를 동시에 달성한 구조적 개선입니다.

------

### 핵심 변화

- `flow-entry`, `flow-state`→ `.flow` 내부 상태 레이어로 통합

반복적으로 발생하는 에너지의 낭비를 막으려는 시도의 예로 파일명과 디렉토리명의 선택이 있습니다.

- `flow-entry/temporary/MEMO.md`
   → `~/my/.flow/core/MEMO.md` → `~/my/.flow/core/memo.md` → `~/my/.flow/memo.md`

Shift키를 누르며 타이핑하는 MEMO.md보다 누르지 않는 memo.md가 더 효율적인 이름입니다.

------

Flow OS는 단순한 디렉토리 설계가 아니라,

> **명령이 아닌 상태를 중심으로 작업하게 만들기 위해**
>
> **구조를 최소화하고 내부로 압축한 시스템입니다.**

결과적으로 줄어든 복잡도 덕분에 (익숙해지면) 사용자는
 **“하나의 Workspace + 하나의 상태”** 만으로 작업을 이어갈 수 있습니다.

## 뇌 과학 기반: CEN ↔ DMN

Flow OS는 뇌의 네트워크 전환 구조에 기반하여 설계되었습니다.

- **CEN (Central Executive Network)**: 목표 지향 행동, 집중, 작업 수행
- **DMN (Default Mode Network)**: 자기 생각, 회상, 상상, mind-wandering
- **SN (Salience Network)**: 중요한 자극 감지, 네트워크 간 전환 조절
- CEN ↔ DMN은 한쪽이 활성화되면 다른 쪽이 억제되는 반대 관계

### 'job'생각과 '잡'생각

- 우선순위 상위 3개 작업(job)과 직접 관련되지 않은 생각을 '잡'생각으로 정의
- '잡'생각은 쓸모없는 것이 아니라, 의미 있는 생각이므로 기록할 가치가 있다
- 다만 'job'생각에 집중하기 위해, 중요한 '잡'생각이라도 일시적으로 흘려보낼 필요가 있다
- 핵심은 '잡'생각을 빠르게 외부에 기록(`memo`)하고, 작업 흐름으로 신속하게 복귀하는 것

### 비유: 평야와 고원

- **DMN = 평야**: 이동이 쉽고 자연스럽지만, 방향 없이 떠다니기 쉽다
- **CEN = 고원**: 진입이 어렵고, 머무르는 것만으로도 에너지가 필요하다
- 인간의 뇌는 기본적으로 평야(DMN)에 머무르기 쉽다
  → 고원(CEN)으로 올라가는 경로를 명확하게 만드는 것이 중요

### 설계 전략

- '잡'생각을 **억제하지 않고**, 빠르게 외부로 분리(memo)하여 작업 상태(CEN)로 복귀
- 중요한 것은 "집중 유지"가 아니라 **"빠른 복귀"**와 **"낮은 재시작 비용"**
- 이탈에는 가드레일을 치고, 이탈 시 재진입의 마찰을 최소화한다

```
이탈 → memo (외부 배출) → todo (현재 위치 확인) → go (즉시 복귀)
```

## ADHD와 Flow OS

> ⚠️ 성인 ADHD를 기준으로 한 인지 흐름 모델이며, 아동 ADHD의 발달적 특성은 포함하지 않습니다.

- ADHD에서는 작업 수행 중에도 DMN이 충분히 억제되지 않아, 내부 사고가 지속적으로 개입할 수 있다 (DMN intrusion)
- SN의 전환 효율이 낮아져, 집중 상태로의 전환 및 유지가 더 많은 인지적 비용을 요구한다
- Flow OS는 이러한 인지적 비용을 줄이고자 하는 바람에서 출발했다

## 왜 만들었는가 (Why?)
개발 환경이 복잡해질수록 겪는 다음의 문제들을 해결하기 위해 설계되었습니다.
- 이전에 하던 작업의 컨텍스트를 떠올리는 데 드는 시간 낭비
- 여러 대의 기기를 사용할 때 작업 상태의 단절
- "정리되지 않은 상태"가 주는 심리적 부담으로 인한 작업 지연

\### 📌 개발 배경 (The Motivation)

중요한 일을 미루는 성향은 모두가 겪는 보편적인 심리적 장애물입니다. 특히 개발 환경을 다시 세팅하는 과정에서 발생하는 마찰(Friction)은 막대한 인지적 에너지를 고갈시킵니다. Flow-os는 이러한 미루기 습관에 대항하는 실천적인 해결책  중 하나가 될 수 있습니다. 작업을 다시 시작할 때의 진입 장벽을 완전히 제거함으로써, 언제든 가볍게 업무로 복귀하고 지속적인 추진력을 얻을 수 있도록 돕습니다.

\> 개인용 미루기 방지 개발 시스템으로 인지적 부하를 최소화해 중요한 작업으로 복귀를 돕고, 몰입 상태 (Flow State)로 들어가는데 도움이 됐으면 하는 바렘으로 설계했습니다.

> 단순한 개발 환경이라기보다 일관되게 작업을 시작, 진행, 마무리하는 방식의 체계이며, 개발 측면에서 뇌가 동작하는 방식을 규격화한 운영 체제에 가까운, 저의 **Personal Development Operating System 2.0를 염두로 시작했습니다. 

\### ⚙️ 핵심 철학 (Core Philosophy)

1. **인지적 부담 완화 (Cognitive Relief):** 준비 단계에서 허비되는 많은 정신적 에너지를 보존하여, 오직 해야 할 일을 실행하는 데만 온전히 집중할 수 있도록 합니다.
2. **마찰 없는 복귀 (Frictionless Return):** 주의 분산을 완벽하게 차단하는 마법 같은 방법은 존재하지 않습니다. 하지만 다시 시작하는 장벽을 획기적으로 낮추면, 다른 길로 빠졌다가도 복귀하기가 훨씬 쉬워집니다. 이는 다시 궤도로 돌아오려는 지속적인 시도를 독려하며, 완전히 손을 놓아버리는 대신 실질적인 진척을 만들어냅니다.
3. **즉각적인 작업 재개 (Instant Resumption):** 워크플로우의 정확한 진입점(Entry point)을 기록합니다. 시작할 때의 마찰이 사라지면 언제든 즉각적이고 끊김 없는 재개가 가능해지며, 궁극적으로는 몰입 상태(Flow State)에 들어가는 습관을 형성합니다.

\### 🧠 작동 원리: 몰입의 심리학 (Why This Works)

**`[ 인지적 부담 완화 ] ➔ [ 마찰 없는 복귀 ] ➔ [ 즉각적인 작업 재개 ]`**

Flow-os의 아키텍처는 습관 형성 이론과 활성화 에너지(Activation Energy)라는 심리학적 배경을 바탕으로 설계되었습니다.

\* 환경 설정에 드는 무거운 정신적 비용을 제거함으로써 (**인지적 부담 완화**), 다시 시작하는 것에 대한 막연한 두려움을 없앱니다.

\* 두려움이 사라지면, 불가피하게 발생한 주의 분산으로부터 다시 복귀하는 과정이 매우 가벼워집니다 (**마찰 없는 복귀**).

\* 시간이 흐르면서 이 쉬운 재진입이 반복되면 행동적 근육 기억(Behavioral muscle memory)이 형성되어, 심리적 저항을 완전히 우회하고 곧바로 업무에 몰입할 수 있게 됩니다 (**즉각적인 작업 재개**).‌
